• Kernel Space vs User Space:

when a program run it basically runs in two spaces 1 - user space 2 - kernal space
User space and kernel space are two separate parts of an operating system that work together to keep the computer both useful and safe. Kernel space is where the kernel, the core part of the operating system, runs with complete access to the computer’s hardware and memory. It can control the CPU, manage RAM, talk directly to devices like disks and network cards, and decide which programs get resources. User space, on the other hand, is where normal applications like your browser, text editor, or terminal run. These programs do not have direct access to hardware or critical system memory because giving them that power would make the system unstable and insecure. Instead, when a program in user space needs to do something important—such as saving a file, opening a webpage, or allocating more memory—it sends a request to the kernel through something called a system call. The kernel checks whether the request is valid and safe, performs the action if allowed, and then sends the result back to the program.

  • Kernel Space: Protected memory area where the core OS, memory management, and hardware drivers run. Full hardware access.
  
  

  • User Space: Restricted memory area where user applications (Python scripts, Web servers, `ls` command) run.

• System Calls (Syscalls): How User Space applications request privileges or hardware access from the Kernel (e.g., `open()`, `read()`, `write()`, `fork()`).
A system call is the mechanism through which a program running in user space requests a service from the kernel (kernel space). Since user-space programs cannot directly access hardware, memory management, or critical system resources, they must ask the kernel to perform these operations on their behalf.

🐧 Linux Day 2/15: Process Management & Signals (SRE Kernel Inspection)
1. What is a Process? (The Infrastructure Runtime)
• Process vs Program: A program is passive code sitting on your disk (e.g., Nginx binary, Python script). A process is that program actively running in the RAM (execution state).

• Process ID (PID): Every running process gets a unique numerical ID assigned by the kernel.

  • `PID 1` is always reserved for the system initialization daemon (`systemd` or `init`), which is the parent of all other processes.

• Process States (Crucial for SRE Debugging):

  • `R` (Running/Runnable): Currently actively utilizing CPU cycles or waiting in the CPU run-queue.

  • `S` (Interruptible Sleep): Waiting for an event or system resource to become available (Most common state).

  • `D` (Uninterruptible Sleep): Process is waiting for critical Hardware I/O (like a slow disk read or network block). Cannot be killed instantly by signals.

  • `Z` (Zombie Process): The process has completed execution but remains in the process table because its Parent process hasn't read its exit status yet. Reclaims memory but holds a PID slot.

---

2. Process Execution Types
• Foreground: Tasks that hijack your terminal session. You must wait for them to finish before entering the next command (e.g., running `ping [suspicious link removed]` directly).

• Background: Running a task without blocking your shell. Append an ampersand `&` to the end of your command (e.g., `python3 automation.py &`). This allows you to keep typing other commands.

• Daemons: System processes that start at boot time and run silently in the background to handle continuous infrastructure requests (e.g., `sshd` for SSH connections, `crond` for cron jobs).

---

3. Core Linux Signals (Inter-Process Communication)
Signals are asynchronous notifications sent by the kernel or a user to tell a running process how to behave.

🛑 The Absolute Mandatory SRE Signals:

1. SIGTERM (Signal 15) - Graceful Shutdown: Asks the process to stop safely. The process will catch this signal, save its state, flush data to logs, close database connections, and exit nicely. (Command: `kill `)

2. SIGKILL (Signal 9) - Hard Force Kill: The kernel forcefully destroys the process instantly. The process cannot catch or ignore this. Warning: It might cause file corruption since it doesn't clean up data. (Command: `kill -9 `)

3. SIGHUP (Signal 1) - Hangup / Dynamic Reload: Used by SREs to tell a background service (like Nginx or Apache) to reload its configuration files without restarting or causing downtime. (Command: `kill -1 `)

---

4. Heavy-Duty SRE Commands for Process Inspection
• `ps aux` or `ps -ef`: Takes a static snapshot of all active processes currently tracked by the kernel.

• `top` / `htop`: Real-time interactive screen showing high CPU/RAM consuming processes dynamically.

• `kill`, `pkill`, `killall`: Commands used to dispatch signals to target PIDs or process names.

• `jobs`, `bg`, `fg`: Local shell utilities to move processes between foreground and background.
