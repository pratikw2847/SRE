• Kernel Space vs User Space:

when a program run it basically runs in two spaces 1 - user space 2 - kernal space
User space and kernel space are two separate parts of an operating system that work together to keep the computer both useful and safe. Kernel space is where the kernel, the core part of the operating system, runs with complete access to the computer’s hardware and memory. It can control the CPU, manage RAM, talk directly to devices like disks and network cards, and decide which programs get resources. User space, on the other hand, is where normal applications like your browser, text editor, or terminal run. These programs do not have direct access to hardware or critical system memory because giving them that power would make the system unstable and insecure. Instead, when a program in user space needs to do something important—such as saving a file, opening a webpage, or allocating more memory—it sends a request to the kernel through something called a system call. The kernel checks whether the request is valid and safe, performs the action if allowed, and then sends the result back to the program.

Like if i am creating a text file so user space dont have direct access to hardware so when i execute command touch it goes from user space to kernal via system call and then kernal interact with hardware and creates a file.

  • Kernel Space: Protected memory area where the core OS, memory management, and hardware drivers run. Full hardware access.
  
  

  • User Space: Restricted memory area where user applications (Python scripts, Web servers, `ls` command) run.

• System Calls (Syscalls): How User Space applications request privileges or hardware access from the Kernel (e.g., `open()`, `read()`, `write()`, `fork()`).
A system call is the mechanism through which a program running in user space requests a service from the kernel (kernel space). Since user-space programs cannot directly access hardware, memory management, or critical system resources, they must ask the kernel to perform these operations on their behalf.

1. What is a Process? (The Infrastructure Runtime)
• Process vs Program: A program is passive code sitting on your disk (e.g., Nginx binary, Python script). A process is that program actively running in the RAM (execution state).
A program is simply a passive set of instructions stored on disk. It is an executable file that contains code written to perform some task.
A process, on the other hand, is an active instance of a program that is currently running in memory.


If you have a shell script file this file is a program. It is just a set of instructions stored on disk. Nothing is happening while it just sits there as a file.
When you execute it Linux creates a process for it. Here’s what happens internally
The shell asks the kernel to run the file using a system call like exec().
The kernel loads the program into memory.
Since it’s a shell script (not raw machine code), Linux starts the interpreter (/bin/bash because of #!/bin/bash).
Bash reads the script line by line and executes those commands.
While it is running, it is a process with its own PID, memory, and CPU time.
When execution finishes, the process ends.

ps aux - it will show cpu utilization and memory utilization as well

3. Core Linux Signals (Inter-Process Communication)
Signals are asynchronous notifications sent by the kernel or a user to tell a running process how to behave.

🛑 The Absolute Mandatory SRE Signals:

1. SIGTERM (Signal 15) — Graceful Shutdown
This is the default signal sent when you use:
kill -15 PID
SIGTERM politely asks the process:

“Please stop now and clean up before exiting.”

The process gets a chance to:

Save unsaved data
Flush logs to disk
Close files safely
Finish active requests
Close database/network connections
Release resources properly

2. SIGKILL (Signal 9) — Force Kill
kill -9 PID
This tells the kernel:

“Destroy this process immediately.”

The kernel instantly removes the process from memory.

The process:

Cannot catch it
Cannot ignore it
Cannot clean up
Cannot save data


3. SIGHUP (Signal 1) — Reload / Hangup
kill -1 PID
“Reload your configuration without stopping.”
This is very useful for production services.
If you update an Nginx config file, instead of restarting Nginx (which could briefly interrupt service), you send:
kill -1 nginx_pid

🐧 Linux & Linux Internals Day 3/15: Advanced Process Monitoring & Troubleshooting Internals
1. Deep Dive: High Load Average & CPU Contention Internals
When an SRE runs `uptime` or `top` on a lagging production server, they see three numbers for Load Average (representing 1, 5, and 15-minute intervals).

• What Load Average Actually Means to the Kernel: It is NOT just CPU usage. The load average is the average number of processes in a runnable state (`R` state, waiting for or using CPU) PLUS processes in an uninterruptible sleep state (`D` state, waiting for disk/network hardware responses).

• CPU-Bound vs. I/O-Bound Load:

  • CPU-Bound: High load average accompanied by high CPU utilization percentages (processes are fighting for CPU time slices).

  • I/O-Bound: High load average but low CPU utilization percentages. This indicates that processes are stuck in `D` state waiting on slow disks or network storage (NFS).

---

2. Kernel Real-Time Performance Analytics Utilities
To debug infrastructure issues, an SRE needs tools that peel back the abstraction layer of the hardware:

A. `top` / `htop` Internals

• Lists active system threads.

• Key Fields to Watch:

  • `VIRT` (Virtual Size): Total memory the process has mapped (including shared libraries and swapped memory).

  • `RES` (Resident Size): The actual physical RAM the process is occupying right now. (Crucial for detecting memory leaks).

  • `%CPU` & `%MEM`: Task resource consumption chunks.

B. `ps` (Process Status) Advanced Flags

• `ps aux`: `a` = show processes for all users, `u` = display user-oriented format, `x` = lift the requirement for a controlling terminal (displays daemons).

• `ps -ef`: `e` = select all processes, `f` = full-format listing (shows Parent Process IDs via `PPID`). Great for tracing parent-child execution loops.

C. `strace` (System Call Tracer) - The SRE Secret Weapon

• What it does: Intercepts and records the system calls called by a process and the signals received by a process.

• SRE Use Case: If an application hangs during initialization, `strace -p ` reveals exactly which system call (like `open()` or `connect()`) it is currently blocked on.

---

3. Managing Process Priorities: `nice` and `renice`
The Linux kernel uses an internal task scheduler to distribute CPU cycles. SREs can influence this scheduler using priority values called Niceness.

• Nice Range: `-20` (Highest priority, least nice to others) to `19` (Lowest priority, very nice). Default value is `0`.

• Commands:

  • `nice -n 10 python3 backup.py`: Starts a background backup script with lower priority so it won't steal CPU cycles from production web traffic.

  • `renice -n -5 -p `: Dynamically escalates an already running process to a higher priority (requires root privileges).
