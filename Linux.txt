• Kernel Space vs User Space:

when a program run it basically runs in two spaces 1 - user space 2 - kernal space
User space and kernel space are two separate parts of an operating system that work together to keep the computer both useful and safe. Kernel space is where the kernel, the core part of the operating system, runs with complete access to the computer’s hardware and memory. It can control the CPU, manage RAM, talk directly to devices like disks and network cards, and decide which programs get resources. User space, on the other hand, is where normal applications like your browser, text editor, or terminal run. These programs do not have direct access to hardware or critical system memory because giving them that power would make the system unstable and insecure. Instead, when a program in user space needs to do something important—such as saving a file, opening a webpage, or allocating more memory—it sends a request to the kernel through something called a system call. The kernel checks whether the request is valid and safe, performs the action if allowed, and then sends the result back to the program.

Like if i am creating a text file so user space dont have direct access to hardware so when i execute command touch it goes from user space to kernal via system call and then kernal interact with hardware and creates a file.

  • Kernel Space: Protected memory area where the core OS, memory management, and hardware drivers run. Full hardware access.
  
  

  • User Space: Restricted memory area where user applications (Python scripts, Web servers, `ls` command) run.

• System Calls (Syscalls): How User Space applications request privileges or hardware access from the Kernel (e.g., `open()`, `read()`, `write()`, `fork()`).
A system call is the mechanism through which a program running in user space requests a service from the kernel (kernel space). Since user-space programs cannot directly access hardware, memory management, or critical system resources, they must ask the kernel to perform these operations on their behalf.

This is the default signal sent when you use:

“Please stop now and clean up before exiting.”

The process gets a chance to:

Close files safely
Finish active requests
Close database/network connections
Release resources properly

2. SIGKILL (Signal 9) — Force Kill
kill -9 PID
This tells the kernel:

“Destroy this process immediately.”

The kernel instantly removes the process from memory.

The process:

Cannot catch it
Cannot ignore it
Cannot clean up
Cannot save data


3. SIGHUP (Signal 1) — Reload / Hangup
kill -1 PID
“Reload your configuration without stopping.”
This is very useful for production services.
If you update an Nginx config file, instead of restarting Nginx (which could briefly interrupt service), you send:

MiB Mem (Physical Memory): This represents your actual, physical RAM (Random Access Memory) hardware installed on the server.
MiB Swap (Virtual Memory): This is a designated space on your Hard Drive or SSD (either a dedicated swap partition or a swap file) that the Linux kernel uses as an extension of your RAM.

The Scenario: Imagine a sudden traffic spike hits your application, or a developer writes a piece of code that causes a temporary memory leak.

Without Swap: The moment your RAM hits 100%, the Linux kernel has no choice but to trigger the OOM (Out Of Memory) Killer. It will immediately terminate your primary process (like your Docker container, Java app, or Database) to keep the operating system alive. Your application goes down completely.

With Swap: Swap acts as an overflow cushion. The system slows down because it's using the disk, but it stays alive. It gives your autoscaling policies time to kick in, or gives you (the SRE) time to log in, investigate, and fix the issue before a total outage occurs.

the Swap space rarely consumes more than 1% to 3% of the total SSD storage.

3. Managing Process Priorities: `nice` and `renice`
The Linux kernel uses an internal task scheduler to distribute CPU cycles. SREs can influence this scheduler using priority values called Niceness.

ps -el command to check nice number

• Nice Range: `-20` (Highest priority, least nice to others) to `19` (Lowest priority, very nice). Default value is `0`.

• Commands:

  • `nice -n 10 python3 backup.py`: Starts a background backup script with lower priority so it won't steal CPU cycles from production web traffic.

  • `renice -n -5 -p `: Dynamically escalates an already running process to a higher priority (requires root privileges).

* Architectural Definition of a Zombie Process (STAT: Z)
A Zombie Process (denoted by Z in ps or top utility outputs) is a process that has completed execution via the exit() system call but still occupies an entry slot inside the kernel's process table.
It is not a running or executable piece of code; it is merely a dead shell. When a process terminates, the operating system releases all of its allocated hardware physical memory (RAM), open file descriptors, and CPU structures

* The SRE Production Risk: PID Slot Exhaustion vs Hardware Resources
* The 'exit()' System Call vs Parent 'wait()' / 'waitpid()' Neglect
The Child Exit: The child process completes its operational loop or encounters an error and triggers the exit() system call. It moves into the TASK_ZOMBIE state.
The Parent Duty: The parent process is expected to continuously execute a blocking or non-blocking wait() or waitpid() system call.


* SIGCHLD Signal Mechanics 
* Resolution Workflows: Finding the PPID, forcing a reap, and parent termination

TOPIC 1: STORAGE LAYER ARCHITECTURE (DISK HARDWARE TO KERNEL)
• Block Devices vs Character Devices
In Linux, the foundational philosophy is: "Everything is a file." The kernel represents hardware components as special files inside the /dev/ (Device) directory. However, based on how data flows between the kernel and the hardware, they are split into two distinct architectures:

Character Devices (char):

The Logic: They stream data with the kernel Byte-by-Byte (sequential stream). Data is processed directly without being cached or stored in a buffer. You cannot jump around randomly; you must read it in a straight line.

Production Examples: * /dev/tty (The terminal/console input)

/dev/urandom (The kernel's random number generator)

Keyboards, mice, and serial ports.

Block Devices:

The Logic: They do not transfer data byte-by-byte. Instead, they move data in fixed-size chunks called "Blocks" (typically 4KB or 8KB). Because data can be buffered (cached) in kernel RAM, the system can perform Random Access—meaning it can instantly jump to any sector on the disk.

Production Examples: All storage media—HDDs, SSDs, NVMe drives, and USB sticks.

• Storage Naming Conventions (/dev/sdX vs /dev/nvmeXnX)

Storage Naming Conventions
When you run system diagnostics using the lsblk command, the Linux kernel displays specific naming prefixes based on the underlying hardware controller and interface technology:

/dev/sdX (SCSI / SATA Storage)
Use Case: Legacy mechanical hard drives (HDDs), standard SATA Solid State Drives (SSDs), and traditional cloud block storage (like older AWS EBS volumes).

Drive Assignment: * First disk detected: /dev/sda

Second disk detected: /dev/sdb

Third disk detected: /dev/sdc ... (and so on).

Partition Sub-naming: When you slice a disk into partitions, the kernel appends a number to the drive string:

/dev/sda1 (Partition 1 on the first disk)

/dev/sda2 (Partition 2 on the first disk)

/dev/nvmeXnX (Non-Volatile Memory Express)
Use Case: Modern, high-throughput NVMe M.2 or PCIe SSDs used in enterprise servers for blazing-fast I/O operations.

Drive Assignment: The naming follows the controller and namespace layout: nvme[Controller_Number]n[Namespace_Number]

First NVMe drive on controller 0: /dev/nvme0n1

Partition Sub-naming: Because the drive name already ends in a number (n1), the kernel injects a p (Partition indicator) before the partition number to avoid confusion:

/dev/nvme0n1p1 (Partition 1 on NVMe drive 0)

/dev/nvme0n1p2 (Partition 2 on NVMe drive 0)

TOPIC 2: FILE SYSTEM INTERNALS & THE POWER OF INODES
• What is a File System? (ext4, XFS)
A raw block device is just a blank sequence of storage sectors. A File System is the structured software layer injected onto a partition that governs how data is stored, organized, named, and retrieved.

ext4 (Fourth Extended Filesystem):

Mechanics: The traditional Linux standard. It uses a fixed allocation scheme—when you format the disk, a predetermined, immutable number of indexes (Inodes) are created.

Best Use Case: Standard Linux distributions, root (/) partitions, and general-purpose workloads.

XFS (Extents-Based Filesystem):

Mechanics: An enterprise-grade, high-performance file system. Unlike ext4, XFS allocates inodes dynamically as files are created (up to a max percentage limits set during initialization).

Best Use Case: Enterprise storage arrays, database clusters, and massive file servers handling multi-terabyte datasets.

• The Inode Architecture & File Metadata Tracking
In Linux, a filename is completely separated from its data. The file name is merely a human-readable alias stored inside a directory index pointing to an Inode (Index Node) number.

An Inode is a fixed 256-byte metadata storage structure inside the file system layer. It does NOT store the filename or the actual data contents. Instead, it tracks:

File Type (Regular file, directory, symlink, block device)

Permissions (Read, Write, Execute flags for User/Group/Others)

Ownership (UID - User ID, GID - Group ID)

Timestamps (mtime - modification, ctime - metadata change, atime - last access)

Size (File size in bytes)

Data Block Pointers: Explicit pointer addresses tracking exactly which hard blocks on the physical hardware contain the raw binary fragments of the file.

TOPIC 4: MOUNT MECHANICS & STATIC MAPPING
• Kernel Mount Concepts (Attaching Hardware to Directories)
The Core Logic: Linux does not use drive letters (like C: or D:). It uses a single unified virtual directory tree starting at root (/). "Mounting" is the process of attaching a formatted partition (e.g., /dev/sdb1) to a specific directory (the "Mount Point") so the kernel can route read/write system calls to the hardware blocks.

mount /dev/sdb1 /mnt/data     # Attaches storage to a directory path
umount /mnt/data             # Safely detaches the storage layer
